# Experiment: exp_08_hyde_hybrid_comparison_UDAY
- **Date:** 2026-05-19
- **Dataset:** Golden Dataset (evaluation/dataset/evaluation_dataset.json)
- **Search Types Compared:** hybrid (baseline) vs HyDE-augmented hybrid
- **Script:** [scripts/hyde_experiment.py](scripts/hyde_experiment.py#L1-L200)

## Objective

Compare baseline hybrid retrieval (vector + pg_trgm RRF fusion) against a HyDE-style
retrieval strategy where short pseudo-answers (hypotheses) are generated by an LLM,
each pseudo-answer is embedded and used to perform retrieval, and results are fused
to increase recall of context that contains the `expected_answer`.

## Hypothesis

HyDE-style pseudo-answer expansion will increase the chance that at least one
pseudo-answer surface relevant documents (improving recall for cases where the
original query poorly matches document phrasing). This should increase the
proportion of queries where the `expected_answer` appears in the final retrieved
context compared to baseline hybrid retrieval.

## Implementation approach

- New experimental script: [scripts/hyde_experiment.py](scripts/hyde_experiment.py#L1-L200).
- Reuses existing components: `src.core.retriever.retrieve_hybrid`, the shared
  embedder in `src.core.pipeline` (`embed_model`), and `src.core.llm.query_llm`.
- For each query in `evaluation/dataset/evaluation_dataset.json` the script:
  1. Runs baseline hybrid retrieval and records top-N document ids.
  2. Requests N pseudo-answers from `query_llm()` (HyDE). If LLM is not configured
     the script falls back to the original query as a single pseudo-answer.
  3. Embeds each pseudo-answer and runs hybrid retrieval; fuse results using a
     simple reciprocal-rank-like aggregation across pseudo-answers.
  4. Checks whether `expected_answer` is present in the fused top-K context
     (substring match — consistent with existing evaluation scripts).
  5. Writes per-query outcomes and aggregate metrics to `experiments/results/`.

## Retrieval flow

Baseline hybrid retrieval:
- Query → embed → `retrieve_hybrid(query, embedding)` → RRF fusion(vec+trgm) → top-K

HyDE retrieval flow:
- Query → generate N pseudo-answers via `query_llm()` → for each pseudo-answer:
  embed → `retrieve_hybrid(pseudo, embedding)` → accumulate reciprocal-rank scores →
  final fused ranking → top-K

Both flows reuse the same `retrieve_hybrid` implementation so the experiment isolates
the effect of pseudo-answer expansion.

## Expected advantages

- Increased recall for queries where surface-level phrasing mismatch prevents
  baseline retrieval from finding relevant documents.
- Robustness to paraphrase and multi-hop phrasing when pseudo-answers capture a
  different lexical form of the target answer.

## Limitations

- Dependence on LLM quality: low-quality pseudo-answers can add noise and
  retrieve irrelevant documents, potentially decreasing precision.
- HyDE increases the number of retrieval calls (HyDE_count × retrievals),
  increasing latency and database load.
- The evaluation uses substring matching against `expected_answer`, which may
  under-count partial matches or semantically equivalent answers.

## Runtime considerations

- HyDE increases calls to the vector/trgm retrieval layer by roughly the number
  of pseudo-answers requested (configurable in the script via `HYDE_COUNT`).
- Expect retrieval time to scale linearly with `HYDE_COUNT` (plus fusion overhead).
- LLM latency/additional cost depends on the configured provider (Groq/OpenRouter/etc.).
- To run at scale, consider batching pseudo-answer generation, caching pseudo-answer
  embeddings, and limiting HyDE to queries that fail baseline recall thresholds.

## Possible failure cases

- LLM provider unavailable or rate-limited → script falls back to query-as-pseudo-answer
  (graceful degradation) but HyDE advantage will be absent.
- Pseudo-answers hallucinate entities leading to retrieval of unrelated documents.
- Very short or ambiguous queries may generate pseudo-answers that are too generic.

## Production implication

- HyDE is useful as an optional recall-boosting layer: enable only when baseline
  retrieval confidence is low, or run asynchronously as a background augmentation
  to reduce latency impact on the user-facing path.
- Consider running HyDE at a lower `HYDE_COUNT` in production with cached
  pseudo-answer embeddings and a fallback policy to avoid high LLM costs.

## Surprising findings (observational / expectations)

- HyDE often helps when answers use domain-specific phrasing not present in the
  query (e.g., legal clause references). However, it can also surface more noise
  for broad analytical queries.
- The balance between recall and precision is sensitive to the HyDE prompt and
  the number of pseudo-answers; small `HYDE_COUNT` (2–4) tends to be a reasonable
  engineering trade-off in practice.

## Conceptual comparison: Baseline hybrid vs HyDE

- Baseline hybrid focuses on matching the original query across vector and
  trigram spaces and performs a single retrieval pass.
- HyDE expands the query space by generating multiple plausible answers/hypotheses,
  each acting as an alternative retrieval key; results are fused to increase the
  chance of surfacing relevant evidence.
- HyDE trades increased compute (LLM + multiple retrievals) for higher recall on
  under-specified or paraphrased queries.

   run in fallback mode.

python3 scripts/hyde_experiment.py
